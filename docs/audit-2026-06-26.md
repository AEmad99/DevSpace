### 2. Sensitive-subpath deny list is broken on Windows (the safety net for your HOME expansion)

**File:** `backend/src/tool_execution.py:74`
**Code:**
```python
def _is_sensitive_path(resolved: str) -> bool:
    parts = resolved.split(os.sep)        # <-- splits by '\' on Windows
```
**Bug:** On Windows, `os.sep == '\\'`. A POSIX-style path like `/home/user/.ssh/authorized_keys` (which an LLM agent will commonly produce) splits into a single element `["/home/user/.ssh/authorized_keys"]`. No component matches `.ssh`, `.gnupg`, `.bashrc`, etc. → **returns False → path is not blocked**.

**Why this matters now:** Your in-progress patch (`scripts/_patch_tool_path_roots.py`, applied to `backend/src/tool_execution.py` via the uncommitted diff) adds **HOME to the default allowlist**, and the comment explicitly relies on `_is_sensitive_path` as the safety net:

> "Sensitive subpaths (.ssh, .gnupg, id_rsa, shell rc files, etc.) are STILL blocked by `_is_sensitive_path` in `_resolve_tool_path`; this is just an allow-list expansion, not a security relaxation."

On Windows that safety net doesn't fire. A prompt-injection asking the agent to `read ~/.ssh/authorized_keys` or `write ~/.bashrc` on a Windows install slips through.

**Failing tests (all of which exercise POSIX-style paths on Windows):**
- `test_sensitive_ssh_dir`
- `test_sensitive_gnupg_dir`
- `test_sensitive_shell_rc`
- `test_sensitive_key_filenames`

**Fix:** split on both `os.sep` AND `/` (or use `pathlib.PurePosixPath`):
```python
def _is_sensitive_path(resolved: str) -> bool:
    parts = re.split(r"[\\/]", resolved)
    ...
```
Add Windows variants of the sensitive basenames if you want defense in depth:
- Windows: `.ssh`, `id_rsa`, etc. live under `%USERPROFILE%` → same deny list works once split is fixed.
- Also worth considering: `NTUSER.DAT`, `Credentials`, `AppData\Local\Microsoft\Credentials`.
